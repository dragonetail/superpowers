# AI 适配格式规范

## 成果物元数据头

所有 Markdown/YAML 成果物必须包含 YAML 元数据头：

```yaml
---
id: [阶段代码]-[序号]           # 必填：唯一标识
name: [成果物名称]              # 必填：成果物名称
version: v1.0                  # 必填：版本号
phase: [阶段名称]               # 必填：所属阶段
status: 草稿                   # 必填：草稿/评审中/已批准/已变更
created_at: YYYY-MM-DD         # 必填：创建日期
updated_at: YYYY-MM-DD         # 必填：更新日期
author: [作者]                 # 必填：创建者
traces_to: []                  # 可选：追溯的上游成果物ID
tags: []                       # 可选：标签
---
```

## ID 编码规则

```
格式：[阶段代码]-[序号]

阶段代码：
- PRJ: 项目启动 (Phase 1)
- BIZ: 业务/需求 (Phase 2)
- ARC: 架构/设计 (Phase 3)
- ENG: 工程/开发 (Phase 4)
- TST: 测试 (Phase 5)
- RLS: 发布 (Phase 6)
- OPS: 运维 (Phase 7)
- RUN: 运营 (Phase 8)
```

## 格式选择指南

| 成果物类型 | 推荐格式 | AI 可读性 | 说明 |
|-----------|---------|-----------|------|
| 业务流程 | BPMN 2.0 XML | ★★★★★ | 标准流程建模语言 |
| 数据模型 | JSON Schema | ★★★★★ | 结构化数据定义 |
| API 规范 | OpenAPI 3.0 | ★★★★★ | API 标准描述 |
| 架构图 | C4 + PlantUML | ★★★★☆ | 文本化架构图 |
| 业务规则 | 决策表 + Markdown | ★★★★★ | 规则明确可追溯 |
| 用户故事 | Markdown + YAML | ★★★★★ | 头部元数据 + 内容 |

## 成果物追溯链示例

```
API设计(ARC-004)
    ← 追溯到 用户故事(BIZ-004)
        ← 追溯到 业务目标(PRJ-001)

测试用例(TST-001)
    ← 追溯到 用户故事(BIZ-004)
        ← 追溯到 业务目标(PRJ-001)

价值度量(RUN-003)
    ← 追溯到 业务目标(PRJ-001)
```

## 文件命名规范

```
格式：[序号]-[成果物名称].[扩展名]

示例：
01-业务目标.yaml
02-干系人分析.md
03-用户故事.md
04-架构设计.md
05-api-design.yaml
06-追溯矩阵.md
```
