# 交付阶段详细规范

## 开发阶段规范

### 代码规范
- 遵循项目既有的编码标准
- 使用 linter 和 formatter 自动化检查
- 代码必须有有意义的注释

### 单元测试要求
- 覆盖率目标：≥80%
- 关键路径必须 100% 覆盖
- 使用 AAA 模式（Arrange-Act-Assert）

### CI/CD 配置
- 每次提交触发构建
- 合并前必须通过所有检查
- 自动化测试套件

## 测试阶段规范

### 测试类型
| 类型 | 覆盖范围 | 执行时机 |
|------|---------|---------|
| 单元测试 | 函数/方法级别 | 每次提交 |
| 集成测试 | 模块间交互 | 合并前 |
| E2E测试 | 完整用户流程 | 发布前 |
| 性能测试 | 关键性能指标 | 发布前 |

### 缺陷分级
| 级别 | 定义 | 处理要求 |
|------|------|---------|
| P0-致命 | 系统崩溃/数据丢失 | 立即修复 |
| P1-严重 | 核心功能不可用 | 24小时内修复 |
| P2-一般 | 功能受限/体验问题 | 发布前修复 |
| P3-轻微 | UI/文案问题 | 可延后 |

## 发布阶段规范

### 发布检查清单
- [ ] 所有测试通过
- [ ] 代码评审完成
- [ ] 文档已更新
- [ ] 回滚方案已准备
- [ ] 监控告警已配置
- [ ] 值班人员已就位

### 发布窗口
- 建议时间：22:00-24:00
- 避免时间：业务高峰期、节假日前

## 运维阶段规范

### 监控指标
| 类型 | 指标 | 告警阈值 |
|------|------|---------|
| 可用性 | 系统可用率 | <99.9% |
| 性能 | 响应时间 | P99 > 500ms |
| 错误 | 错误率 | >0.1% |
| 资源 | CPU/内存 | >80% |

### 故障处理流程
1. 告警触发 → 2. 初步定位 → 3. 影响评估 → 4. 应急处理 → 5. 根因分析 → 6. 预防措施

## 运营阶段规范

### 业务指标追踪
- 追溯到 PRJ-001 定义的业务目标
- 定期（周/月）生成追踪报告
- 异常指标需要分析原因

### 价值度量报告
- 包含：KPI达成情况、用户反馈、系统运行状态
- 输出：迭代建议、下一轮启动输入

### 闭环反馈
运营阶段的输出直接作为下一轮 ai-sldc-design 的输入：

```
RUN-003 价值度量报告 → 下一轮 PRJ-001 业务现状回顾
RUN-002 用户反馈分析 → 下一轮 BIZ-004 改进机会
RUN-004 迭代建议    → 下一轮 PRJ-001 新业务目标
```
