# GTMC 软件工程成果物与资产体系研讨

> 研讨日期：2026-03-22
> 研讨背景：基于现有 `/superpowers:brainstorming` skill 扩展，结合公司软件工程规范，建立成果物规范管理体系

---

## 一、研讨背景与目标

### 1.1 初始需求

- 现有 `/superpowers:brainstorming` skill 体系和内容很好，但希望扩展以支持：
  - 需求、产品、设计、开发各阶段的成果物规范管理
  - 结合公司软件工程规范

### 1.2 现状分析

- **规范现状**：没有体系化的规范和约定，很多知识在资深老师的脑子中，需要通过收集和交互确认
- **知识状态**：隐性知识为主，需要显性化

### 1.3 目标需求（ABCD全要）

| 编号 | 需求 | 说明 |
|------|------|------|
| A | 标准化输出 | 每次头脑风暴后能自动生成符合规范的文档 |
| B | 阶段间衔接 | 需求→产品→设计→开发 各阶段成果物之间有清晰的追溯和关联 |
| C | 质量把关 | 每个阶段的成果物有检查清单/评审标准 |
| D | 知识沉淀 | 将老师们的经验固化成模板，让新人也能产出高质量成果物 |

## 二、方案路线选择

### 2.1 三种路线方案

#### 路线 1：渐进式模板收集模式

在现有 brainstorming 流程中嵌入"知识收集点"——每次使用时，如果遇到缺失的规范，就引导用户去询问老师并记录下来。

- **优点**：不需要前期大量投入，边用边建
- **缺点**：初期产出质量可能不稳定，依赖使用频率

#### 路线 2：先建框架后填充模式

先设计一个完整的"成果物规范体系框架"，定义清楚各阶段应该产出什么、有什么检查项。框架先留空或用通用模板。

- **优点**：结构清晰，有章可循
- **缺点**：前期需要投入时间设计框架

#### 路线 3：老师访谈优先模式

先系统地访谈老师们，把隐性知识显性化，形成初版规范文档。

- **优点**：一次性把基础打牢
- **缺点**：访谈整理工作量大

### 2.2 最终选择

**组合方案**：路线 2 + 路线 3 + 试点路线 1

```
框架设计 → 关键知识提取 → 试点迭代完善
```

## 三、Skill 架构设计

### 3.1 三种架构方案

#### 方案 A：增强 brainstorming

直接在现有 SKILL.md 中增加阶段管理、成果物模板、检查清单等功能。

| 优点 | 缺点 |
|------|------|
| 一个 skill 解决所有问题 | brainstorming 会变得臃肿 |
| 不需要切换 skill | 违背"单一职责"原则 |

#### 方案 B：新建独立 skill

创建新 skill 如 `/superpowers:spec-workflow`，brainstorming 完成后手动衔接。

| 优点 | 缺点 |
|------|------|
| 职责分离清晰 | 需要管理两个 skill 衔接 |
| brainstorming 保持简洁 | 用户需要记住两个命令 |

#### 方案 C：模块化组合

brainstorming 保持原样 + 新建成独立的"成果物管理器"skill + 两者通过约定接口配合。

| 优点 | 缺点 |
|------|------|
| 最大灵活性 | 设计复杂度最高 |
| 长期可维护性最好 | 初期投入较大 |

### 3.2 方案 B 与 C 的区别

```
方案 B：两个独立的 Skill
┌──────────────┐      ┌──────────────────┐
│ brainstorming │ ───→ │ spec-workflow    │
│              │      │                  │
│ 独立运行      │      │ 独立运行          │
└──────────────┘      └──────────────────┘
关系：两个平行的 skill，靠用户手动衔接
本质：1+1 = 2 个独立工具

方案 C：模块化组合（有"胶水"层）
┌──────────────┐      ┌─────────────┐      ┌─────────────┐
│ brainstorming │ ───→ │ 成果物核心   │ ←─── │ 检查清单模块 │
│              │      │ 模块        │      │             │
└──────────────┘      └─────────────┘      └─────────────┘
                            ↑
                    ┌───────┴───────┐
                    │   协调层       │
                    └───────────────┘
关系：有共享的"核心模块"，各 skill 调用同一套
本质：1+1+核心 = 有机整体
```

### 3.3 最终选择

**先采用方案 B，快速出效果，后期再向 C 演进**

### 3.4 新 Skill 命名

**确定名称**：`/superpowers:spec-workflow`

## 四、原始成果物体系

### 4.1 原始规划（来自用户提供）

**GTMC数字化SLDC成果物理想状态 — AI时代的数字化知识资产体系**

基于 Gartner 软件工程体系、数字化转型实践及 AI-Coding 技术，遵循 **【业务→架构→设计→代码→配置→KPI】** 的知识流动闭环。

| 阶段 | 标题 | 成果物 | 目标 |
|------|------|--------|------|
| (A) | 业务知识 | 业务目标/KPI、业务流程模型(BPMN)、业务对象模型、业务规则(SOP)、业务场景/用户故事、业务能力与事件模型、原型设计 | 让AI/人理解业务，删除/弱化非结构化需求文档 |
| (B) | 架构知识 | 业务能力模型、领域边界划分、系统上下文图、应用架构图、系统组件图、业务主流程交互时序 | 让AI/人理解系统结构 |
| (C) | 设计知识 | API设计、数据模型、系统交互时序、关键业务规则、非功能需求要点 | 让AI理解业务并生成高质量代码 |
| (D) | 工程实现知识 | 开发规范/代码规范、源代码、单元测试、构建脚本、CI/CD配置、环境配置 | 让AI维护/使用 |
| (E) | 测试验证知识 | 缺陷知识库、测试用例、自动化测试、测试报告 | 实现测试自动化 |
| (F) | 运行运维知识 | 配置资产库、系统监控指标、故障知识库、运行事件记录、业务指标 | 实现运维自动化 |
| (G) | 项目治理/过程知识 | 需求评估报告、项目决策报告、项目立项资料、项目计划书、业务要件评审结论、架构评审结论、合规检查记录、业务试行报告 | 面向业务闭环治理 |

## 五、优化建议

### 5.1 阶段边界问题

**现状**：G（项目治理）放在最前面，但治理活动其实是贯穿全流程的

**建议**：把 G 拆分
- G1. 项目启动 → 放在规划阶段（立项、计划）
- G2. 阶段评审 → 分布到各阶段末尾（业务评审、架构评审、设计评审）
- G3. 项目收尾 → 放在运维阶段后（试行报告、复盘）

### 5.2 成果物颗粒度不均

| 阶段 | 现有数量 | 建议调整 |
|------|---------|---------|
| A. 业务 | 7 | 可考虑合并"业务对象模型"+"业务能力与事件模型" |
| C. 设计 | 5 | 可补充"接口契约"、"安全设计"等 |

### 5.3 缺少的关键成果物

| 阶段 | 缺失的成果物 | 理由 |
|------|-------------|------|
| A. 业务 | 用户画像/干系人分析 | 理解"为谁做" |
| B. 架构 | 技术选型决策 | AI需要知道技术约束 |
| C. 设计 | 变更影响分析 | 支持迭代场景 |
| D. 工程 | 代码审查记录 | 知识沉淀 |
| E. 测试 | 性能基准 | 可量化对比 |

### 5.4 AI协作适配性

**现状**：强调"结构化"，但没说明用什么格式

**建议**：明确各成果物的推荐格式
```
结构化格式建议：
├─ 流程类 → BPMN/流程图JSON
├─ 模型类 → JSON Schema / YAML
├─ API类 → OpenAPI Spec
├─ 规则类 → 决策表/规则引擎格式
└─ 文档类 → Markdown + 结构化模板
```

### 5.5 阶段间追溯关系

**建议**：增加成果物ID和关联机制
```
例：API设计(ID: API-001)
    ← 追溯到 业务场景(ID: USC-003)
    ← 追溯到 业务规则(ID: SOP-012)
```

## 七、最终体系设计

### 7.1 术语统一

| 概念 | 建议术语 | 英文 | 说明 |
|------|---------|------|------|
| 时间维度 | **阶段** | Phase | 如：规划阶段、开发阶段 |
| 活动类型 | **过程** | Process | 如：需求过程、设计过程 |
| 工作产出 | **成果物** | Artifact | 过程产出的文档、代码、配置等 |
| 沉淀价值 | **资产** | Asset | 可复用的结构化成果物 |
| 质量关卡 | **评审点** | Review Gate | 阶段间的质量门禁 |

### 7.2 核心术语对照表

| 中文术语 | 英文术语 | 定义 |
|---------|---------|------|
| **阶段** | Phase | 生命周期的时间切分，如"需求阶段" |
| **过程** | Process | 特定目标的系列活动，如"需求定义过程" |
| **成果物** | Artifact | 过程产出的具体内容 |
| **资产** | Asset | 可复用的结构化成果物 |
| **评审点** | Review Gate | 阶段间的质量门禁 |
| **追溯** | Traceability | 成果物之间的关联关系 |
| **资产化** | Assetization | 成果物转化为资产的过程 |
