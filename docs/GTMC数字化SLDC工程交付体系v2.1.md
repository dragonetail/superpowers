# 第一章：体系概述

## 1.1 背景与目标

### 1.1.1 背景

GTMC数字化SLDC（软件生命周期开发）工程交付体系旨在建立AI时代的数字化知识资产体系，基于以下理念：

- **业务驱动工程**：Business-Driven Engineering，业务知识是软件开发的核心驱动力
- **AI协作适配**：让AI能够理解、生成、复用企业数字化知识
- **知识流动闭环**：从业务价值定义到价值度量反馈的完整闭环

### 1.1.2 目标

| 目标 | 说明 |
|------|------|
| 标准化交付 | 建立统一的工程交付标准和流程 |
| 质量可控 | 每个阶段有明确的准入准出条件 |
| AI友好 | 成果物格式适配AI理解和处理 |
| 知识沉淀 | 将隐性知识转化为可复用的数字化资产 |

### 1.1.3 知识流动闭环

```
                    ┌─────────────────────────────────────────┐
                    │           GTMC 知识流动闭环              │
                    └─────────────────────────────────────────┘
                                       │
                                       ▼
                            ┌──────────────────┐
                            │  业务价值定义     │ ← 1.启动
                            │  (为什么做)       │
                            └────────┬─────────┘
                                     │
                                     ▼
                            ┌──────────────────┐
                            │  需求要件整理     │ ← 2.需求
                            │  (做什么)         │
                            └────────┬─────────┘
                                     │
                                     ▼
                            ┌──────────────────┐
                            │  技术方案设计     │ ← 3.设计
                            │  (怎么做)         │
                            └────────┬─────────┘
                                     │
                                     ▼
                            ┌──────────────────┐
                            │  代码实现交付     │ ← 4.开发+5.测试+6.发布
                            │  (做出来)         │
                            └────────┬─────────┘
                                     │
                                     ▼
                            ┌──────────────────┐
                            │  系统运行验证     │ ← 7.运维
                            │  (跑起来)         │

                                     │

                            ┌──────────────────┐
                            │  价值度量反馈     │ ← 8.运营
                            │  (效果如何)       │
                            └────────┬─────────┘
                                     │
                                     │  持续改进
                                     └─────────────────→ 回到业务价值定义
```

## 1.3 行业对齐与视角

### 1.3.1 行业标准参考

| 标准/框架 | 核心视角 | 可借鉴点 |
|-----------|---------|---------|
| **ISO/IEC 12207** | 系统生命周期过程 | 过程分类：协议、组织、项目、技术 |
| **PMBOK** | 项目管理 | 过程组：启动→规划→执行→监控→收尾 |
| **SAFe** | 敏捷规模化 | 需求层次：Epic→Feature→Story |
| **DevOps** | 工程实践 | 流水线：Plan→Code→Build→Test→Release→Operate |

### 1.3.2 三个核心视角

本体系从三个视角进行组织和管理：

```
┌─────────────────────────────────────────────────────────────────────────┐
│                                                                         │
│    【工程视角】              【管理视角】              【资产视角】      │
│    Engineering View         Governance View           Asset View       │
│    ─────────────            ─────────────             ──────────       │
│    "做什么、怎么做"          "管什么、控什么"          "存什么、用什么"   │
│                                                                         │
│    技术活动与产出            决策与质量把控            知识沉淀与复用     │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

**视角说明：**

| 视角 | 关注点 | 典型活动 | 产出 |
|------|--------|---------|------|
| 工程视角 | 技术实现 | 需求分析、架构设计、编码、测试 | 成果物（Artifact） |
| 管理视角 | 质量控制 | 评审、审批、监控、决策 | 评审记录、决策文档 |
| 资产视角 | 知识复用 | 沉淀、标准化、检索、复用 | 数字化资产（Asset） |

### 1.4.1 核心术语定义

| 关联关系 | **追溯** | Traceability | 成果物之间的关联关系 |

## 2.1 阶段1：启动 (Init) —— 业务价值定义

### 【阶段定位】

| 维度 | 说明 |
|------|------|
| 知识流动闭环 | **业务价值定义** — 明确"为什么做" |
| 上游阶段 | 无（起点） |
| 下游阶段 | 2.需求 |

### 【阶段目标】

明确项目的业务价值、目标、范围和可行性，完成立项审批，确保项目"值得做"。

### 【准入条件】

| 条件 | 说明 |
|------|------|
| 业务诉求明确 | 有清晰的业务痛点或机会 |
| 资源可用性 | 关键资源（人、财、时）初步可保障 |
| 发起人支持 | 有明确的业务发起人和支持者 |

### 【工程过程】

| 过程 | 核心活动 | 关键产出 |
|------|---------|---------|
| 项目启动 | 明确业务目标、识别干系人、评估可行性 | 项目章程、干系人登记 |
| 计划编制 | 定义范围、估算资源、制定进度 | 项目计划书 |

### 【成果物清单】

| ID | 名称 | 说明 | AI适配格式 |
|----|------|------|-----------|
| PRJ-001 | 业务目标/KPI | 业务目标与关键成功指标 | YAML + Markdown |
| PRJ-002 | 干系人分析 | 干系人识别与利益分析 | Markdown + YAML |
| PRJ-003 | 需求评估报告 | 业务需求分析与可行性评估 | Markdown |
| PRJ-004 | 项目决策报告 | 项目关键决策记录（ADR格式） | Markdown |
| PRJ-005 | 项目计划书 | 范围、进度、资源、风险计划 | Markdown + YAML |

### 【管理过程】

| 评审点 | 评审内容 | 通过标准 |
|--------|---------|---------|
| 立项评审 | 业务价值、可行性、资源保障 | 业务价值清晰、可行性确认、资源到位 |

### 【准出条件 & 检查清单】

- [ ] 业务目标/KPI已定义并获得认同
- [ ] 干系人已识别并分析
- [ ] 可行性评估已完成

# 第三章：成果物规范（面向AI时代）

## 3.1 资产化四要素

成果物转化为可复用的数字化资产，需满足以下四个条件：

| 条件 | 说明 | 检查要点 |
|------|------|---------|
| **结构化** | 使用标准格式 | 是否使用JSON/YAML/Markdown等结构化格式 |
| **可追溯** | 有唯一标识 | 是否有ID、版本号、关联关系 |
| **可复用** | 可跨项目使用 | 是否已脱敏、是否足够通用 |
| **可检索** | 支持AI检索 | 是否存入知识库、是否有元数据 |

### 资产化成熟度评估

| 等级 | 结构化 | 可追溯 | 可复用 | 可检索 |
|------|--------|--------|--------|--------|
| L1-初始 | Word/Excel | 无ID | 项目内 | 本地存储 |
| L2-基础 | Markdown | 有ID | 团队内 | 文档库 |
| L3-标准 | YAML/JSON | 有追溯链 | 组织内 | 知识库 |
| L4-智能 | 机器可读格式 | 自动追溯 | 跨组织 | AI知识库 |

## 3.4 追溯机制

```
- BIZ-005: 需求阶段的第5个成果物
```

```
    ← 追溯到 用户故事(BIZ-004)

    ← 追溯到 业务目标(PRJ-001)
```

```
    ↓ 提交评审

    ↓ 评审通过          ↓ 评审不通过

    ↓ 需要变更

    ↓ 重新评审
评审中(In Review)

- 评审中：已提交，等待评审
```
