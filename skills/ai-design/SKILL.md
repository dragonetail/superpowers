# AI Design - AI时代的工程化设计

## 概述

基于 **GTMC 数字化 SLDC 工程交付体系 v2.1** 的工程化设计流程，帮助将想法转化为符合规范的结构化设计文档。

通过协作式对话，依次推进 **启动 → 需求 → 设计** 三个阶段，产出 AI 可理解、可复用的数字化成果物。

**知识流动闭环对齐：**
```
业务价值定义 → 需求要件整理 → 技术方案设计 → [交接给 ai-delivery]
```

---

## 核心特性

与基础 brainstorming 相比，ai-design 增加：

| 特性 | 说明 |
|------|------|
| 阶段对齐 | 严格对齐 v2.1 体系的3个阶段 |
| 准入准出 | 每个阶段有明确的准入/准出条件检查 |
| 成果物规范 | 使用 v2.1 规定的 AI 适配格式 |
| 追溯机制 | 自动建立成果物之间的追溯关系 |
| 模板生成 | 按模板生成结构化文档 |

---

## 使用场景

- 需要规范化交付的新功能设计
- 需要产出可追溯成果物的项目
- 需要后续交接给开发团队的设计
- 需要 AI 参与生成成果物的场景

**不适用场景：**
- 快速头脑风暴（请使用 `/superpowers:brainstorming`）
- 简单问题讨论
- 不需要文档输出的探索

## 阶段1：启动 (Init)

### 目标
明确项目的业务价值、目标、范围和可行性，确保项目"值得做"。

### 准入条件
- [ ] 有业务诉求或机会
- [ ] 有关键干系人支持

### 协作流程

**1. 理解背景**
- 了解当前项目上下文（文件、文档、提交记录）
- 询问业务背景和痛点

**2. 明确目标**
- 一次问一个问题，逐步明确业务目标
- 使用多选题引导用户思考
- 聚焦：目的、约束、成功标准

**3. 探索方案**
- 提出 2-3 种不同的方案选择
- 给出推荐方案和理由
- 讨论权衡取舍

### 成果物产出

| ID | 名称 | 格式 | 说明 |
|----|------|------|------|
| PRJ-001 | 业务目标/KPI | YAML + Markdown | 业务目标与关键成功指标 |
| PRJ-002 | 干系人分析 | Markdown + YAML | 干系人识别与利益分析 |
| PRJ-003 | 项目计划书 | Markdown + YAML | 范围、进度、资源计划 |

### 准出检查

- [ ] 业务目标/KPI 已定义并获得认同
- [ ] 干系人已识别并分析
- [ ] 项目计划已编制
- [ ] 用户确认可以进入下一阶段

---

## 阶段2：需求 (Require)

### 目标
完整理解业务，将隐性业务知识转化为结构化需求。

### 准入条件
- [ ] 阶段1 准出检查通过
- [ ] 业务目标已确认

### 协作流程

**1. 业务调研**
- 询问业务流程
- 了解业务规则和约束
- 识别用户角色和场景

**2. 需求分析**
- 编写用户故事（Gherkin 格式）
- 绘制业务流程（BPMN 格式）
- 定义业务对象（JSON Schema）

**3. 需求确认**
- 逐段呈现需求设计
- 每段 200-300 字
- 确认后继续下一段

### 成果物产出

| ID | 名称 | 格式 | 说明 |
|----|------|------|------|
| BIZ-001 | 业务流程模型 | BPMN 2.0 XML | 业务流程图 |
| BIZ-002 | 业务对象模型 | JSON Schema | 业务实体与数据定义 |
| BIZ-003 | 业务规则 | Markdown + 决策表 | 标准操作流程与规则 |
| BIZ-004 | 用户故事 | Markdown + YAML | 用户故事与验收标准 |
| BIZ-005 | 原型设计 | Figma/Sketch 链接 | UI/UX 原型 |

### 准出检查

- [ ] 业务流程模型已完成（BPMN 格式）
- [ ] 业务对象模型已定义（JSON Schema）
- [ ] 用户故事已完成
- [ ] 需求追溯关系已建立
- [ ] 用户确认可以进入下一阶段

---

## 阶段3：设计 (Design)

### 目标
完成架构设计和技术方案，产出可指导开发的结构化设计资产。

### 准入条件
- [ ] 阶段2 准出检查通过
- [ ] 需求已确认

### 协作流程

**1. 架构设计**
- 确定系统上下文
- 划分领域边界（DDD）
- 选择技术栈（记录 ADR）

**2. 详细设计**
- 设计 API（OpenAPI 格式）
- 设计数据模型
- 绘制时序图

**3. 设计呈现**
- 逐段呈现设计
- 每段 200-300 字
- 确认后继续

### 成果物产出

| ID | 名称 | 格式 | 说明 |
|----|------|------|------|
| ARC-001 | 系统上下文图 | C4 Model / PlantUML | 系统边界与外部交互 |
| ARC-002 | 架构图 | C4 Model / PlantUML | 应用架构与组件关系 |
| ARC-003 | 技术选型决策 | Markdown (ADR) | 技术栈选择与理由 |
| ARC-004 | API 设计 | OpenAPI 3.0 | 接口规范 |
| ARC-005 | 数据模型 | ER图 + JSON Schema | 数据库设计 |

### 准出检查

- [ ] 架构图已完成
- [ ] 技术选型决策已记录（ADR 格式）
- [ ] API 设计已完成（OpenAPI 格式）
- [ ] 数据模型已设计
- [ ] 设计追溯关系已建立
- [ ] 用户确认设计完成

**提交到 Git：**
- 将设计文档提交到版本控制
- 使用规范的 commit message

- 确保所有成果物已保存

## AI 适配格式规范

### 成果物元数据头

所有成果物必须包含 YAML 元数据头：

```yaml
---
id: [阶段代码]-[序号]
name: [成果物名称]
version: v1.0
phase: [阶段名称]
status: 草稿
created_at: YYYY-MM-DD
updated_at: YYYY-MM-DD
author: [作者]
traces_to: [上游ID列表]
tags: [标签列表]
---
```

### 格式选择

| 成果物类型 | 推荐格式 | AI 可读性 |
|-----------|---------|-----------|
| 业务流程 | BPMN 2.0 XML | ★★★★★ |
| 数据模型 | JSON Schema | ★★★★★ |
| API 规范 | OpenAPI 3.0 | ★★★★★ |
| 架构图 | C4 Model + PlantUML | ★★★★☆ |
| 业务规则 | 决策表 / Markdown | ★★★★★ |
| 用户故事 | Markdown + YAML | ★★★★★ |

## 追溯机制

### ID 编码规则

```
格式：[阶段代码]-[序号]

阶段代码：
- PRJ: 项目启动
- BIZ: 业务/需求
- ARC: 架构/设计
```

### 追溯链示例

```
API设计(ARC-004)
    ← 追溯到 用户故事(BIZ-004)
        ← 追溯到 业务目标(PRJ-001)
```

## 关键原则

- **阶段推进** — 严格按阶段推进，完成准入准出检查
- **一次一问** — 每次只问一个问题，避免信息过载
- **多选优先** — 优先使用多选题引导思考
- **分段呈现** — 设计分段呈现，每段确认后继续
- **YAGNI 原则** — 坚决移除不必要的功能
- **AI 适配** — 所有产出使用 AI 友好的格式
- **可追溯性** — 自动建立成果物追溯关系
