# AI Delivery - AI时代的工程化交付

## 概述

基于 **GTMC 数字化 SLDC 工程交付体系 v2.1** 的工程化交付流程，承接 `/superpowers:ai-design` 的设计成果，完成从代码实现到价值闭环的完整交付。

通过阶段化管理，依次推进 **开发 → 测试 → 发布 → 运维 → 运营** 五个阶段，确保高质量交付和持续价值反馈。

**知识流动闭环对齐：**
```
[承接 ai-design: 业务价值定义 → 需求要件整理 → 技术方案设计]
    → 代码实现交付 → 系统运行验证 → 价值度量反馈 → [闭环回到业务价值定义]
```

---

## 核心特性

| 特性 | 说明 |
|------|------|
| 阶段衔接 | 无缝承接 ai-design 的设计成果 |
| 阶段推进 | 5个阶段依次推进，每阶段有准入准出检查 |
| 成果物管理 | 按阶段生成和管理成果物 |
| 追溯维护 | 保持与设计阶段的追溯关系 |
| 闭环反馈 | 运营阶段输出反馈到下一轮启动 |

---

## 使用场景

- 承接 ai-design 完成的设计，进入开发阶段
- 管理从开发到运营的完整交付流程
- 需要规范化成果物管理的项目交付
- 需要建立价值闭环反馈的项目

**前置条件：**
- 已完成 `/superpowers:ai-design` 或有等效的设计文档
- 设计阶段成果物已确认

## 阶段4：开发 (Develop)

### 目标
按照设计规范完成代码实现，确保代码质量和可维护性。

### 准入条件
- [ ] 设计已确认（ai-design 阶段3 准出通过）
- [ ] 开发环境就绪
- [ ] 任务已分解

### 协作流程

**1. 接收设计**
- 加载 ai-design 阶段的成果物
- 确认追溯关系

**2. 开发指导**
- 询问开发进度和问题
- 提供技术建议
- 确保代码符合规范

**3. 代码评审准备**
- 检查代码质量
- 确认单元测试覆盖

### 成果物产出

| ID | 名称 | 格式 | 说明 |
|----|------|------|------|
| ENG-001 | 源代码 | Git 仓库 | 项目源码 |
| ENG-002 | 单元测试 | 测试框架格式 | 单元测试代码 |
| ENG-003 | 代码规范 | Markdown | 编码标准与约定 |
| ENG-004 | 技术文档 | Markdown | 技术说明文档 |
| ENG-005 | CI 配置 | YAML | 持续集成配置 |

### 准出检查

- [ ] 功能代码已完成
- [ ] 单元测试覆盖率达标
- [ ] 代码评审已通过
- [ ] CI 构建通过
- [ ] 技术文档已更新

---

## 阶段5：测试 (Test)

### 目标
完成测试验证，确保产品质量达到发布标准。

### 准入条件
- [ ] 阶段4 准出检查通过
- [ ] 代码已完成并通过评审
- [ ] 测试环境就绪

### 协作流程

**1. 测试规划**
- 确认测试范围
- 确认测试用例

**2. 测试执行**
- 跟踪测试进度
- 记录缺陷
- 验证修复

**3. 测试报告**
- 汇总测试结果
- 评估发布风险

### 成果物产出

| ID | 名称 | 格式 | 说明 |
|----|------|------|------|
| TST-001 | 测试用例 | Gherkin / Markdown | 功能/非功能测试用例 |
| TST-002 | 自动化测试 | 测试框架格式 | 自动化测试脚本 |
| TST-003 | 测试报告 | Markdown | 测试结果汇总 |
| TST-004 | 缺陷记录 | 缺陷管理工具 | 缺陷跟踪记录 |

### 准出检查

- [ ] 测试用例已执行
- [ ] 严重缺陷已修复
- [ ] 测试覆盖率达标
- [ ] 性能测试通过
- [ ] 测试报告已完成

---

## 阶段6：发布 (Release)

### 目标
完成部署和上线，确保系统平稳发布。

### 准入条件
- [ ] 阶段5 准出检查通过
- [ ] 测试已通过
- [ ] 发布包已准备

### 协作流程

**1. 发布准备**
- 确认发布内容
- 准备发布检查清单
- 确认回滚方案

**2. 发布执行**
- 执行部署步骤
- 验证部署结果

**3. 发布确认**
- 冒烟测试
- 监控确认

### 成果物产出

| ID | 名称 | 格式 | 说明 |
|----|------|------|------|
| RLS-001 | 部署配置 | YAML / Helm | 部署清单与配置 |
| RLS-002 | 发布检查清单 | Markdown + YAML | 发布前检查项 |
| RLS-003 | 变更记录 | Markdown | 版本变更说明 |
| RLS-004 | 发布报告 | Markdown | 发布结果报告 |

### 准出检查

- [ ] 部署配置已检查
- [ ] 发布检查清单已完成
- [ ] 部署已执行
- [ ] 冒烟测试已通过
- [ ] 监控告警已确认
- [ ] 发布报告已完成

---

## 阶段7：运维 (Operate)

### 目标
保障系统稳定运行，及时处理故障。

### 准入条件
- [ ] 阶段6 准出检查通过
- [ ] 系统已发布
- [ ] 监控已配置

### 协作流程

**1. 运维建立**
- 确认监控覆盖
- 建立运维手册
- 建立故障处理流程

**2. 运行监控**
- 监控系统状态
- 分析运行数据
- 处理告警

**3. 持续优化**
- 性能调优
- 容量规划

### 成果物产出

| ID | 名称 | 格式 | 说明 |
|----|------|------|------|
| OPS-001 | 监控配置 | YAML / JSON | 监控指标与告警规则 |
| OPS-002 | 运维手册 | Markdown | 运维操作指南 |
| OPS-003 | 故障知识库 | 知识库工具 | 故障记录与解决方案 |
| OPS-004 | 运维报告 | Markdown | 运维状态报告 |

### 准出检查

- [ ] 监控覆盖完整
- [ ] 运维手册已编写
- [ ] 故障处理流程已建立
- [ ] 系统运行稳定

---

## 阶段8：运营 (Run)

### 目标
实现业务价值度量，收集反馈，形成闭环。

### 准入条件
- [ ] 阶段7 准出检查通过
- [ ] 系统稳定运行
- [ ] 业务指标定义完成

### 协作流程

**1. 数据采集**
- 确认业务指标埋点
- 收集用户反馈

**2. 价值分析**
- 追踪业务 KPI
- 分析用户行为
- 评估业务价值

**3. 反馈输出**
- 形成价值度量报告
- 输出迭代建议
- 传递到下一轮启动

### 成果物产出

| ID | 名称 | 格式 | 说明 |
|----|------|------|------|
| RUN-001 | 业务指标追踪 | Markdown + 数据 | 业务 KPI 持续追踪 |
| RUN-002 | 用户反馈分析 | Markdown | 用户反馈收集与分析 |
| RUN-003 | 价值度量报告 | Markdown | 业务价值实现度量 |
| RUN-004 | 迭代建议 | Markdown | 基于反馈的迭代建议 |

### 准出检查

- [ ] 业务指标已追踪
- [ ] 用户反馈已收集
- [ ] 价值度量报告已完成
- [ ] 迭代建议已输出
- [ ] 闭环反馈已准备

## 闭环反馈

### 完成运营阶段后

**闭环到下一轮启动：**
```
运营阶段输出                    下一轮启动输入
─────────────                  ─────────────
价值度量报告         →         业务现状回顾
用户反馈分析         →         改进机会识别
迭代建议             →         新的业务目标
```

**触发下一轮：**
- 询问："运营阶段已完成，是否开始下一轮迭代？"
- 如果是，建议使用 `/superpowers:ai-design` 开始新的设计周期

## 阶段状态管理

### 状态流转

```
待开始 (Pending)
    ↓ 开始阶段
进行中 (In Progress)
    ↓ 完成准出检查
已完成 (Completed)
    ↓ 进入下一阶段
待开始 (Pending)
```

### 进度跟踪

每个阶段完成后，记录：
- 阶段名称和状态
- 准入准出检查结果
- 成果物清单
- 遇到的问题和解决方案

---

## 关键原则

- **阶段衔接** — 严格按阶段推进，完成准入准出检查
- **追溯维护** — 保持与设计阶段的追溯关系
- **质量把关** — 每个阶段有明确的质量标准
- **闭环思维** — 运营阶段必须输出反馈
- **AI 协作** — 充分利用 AI 辅助各阶段工作
- **持续改进** — 基于反馈持续优化
